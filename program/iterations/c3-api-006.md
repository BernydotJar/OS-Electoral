## Acceptance criteria

1. `POST /api/v1/tenants/{tenant_id}/campaigns` requires authentication, fresh server-owned tenant authorization and a non-empty `Idempotency-Key`.
2. Authorization matches exact `tenant`, `campaign=null`, `workspace=null`, `action=create`, `resource_type=campaign_collection`, `resource_id=<tenant_id>`, `purpose=Create tenant campaign`, validity and revocation state before persistence is invoked.
3. The request accepts bounded slug, name, jurisdiction and stage fields only; text is normalized and whitespace-only values are rejected.
4. The server owns campaign ID, `status=DRAFT`, `version=1`, timestamps and all evidence identifiers.
5. Campaign, audit event, internal outbox event and idempotency record commit atomically under forced tenant RLS.
6. Same key plus same normalized request and authority replays the exact committed evidence without duplicates; same key plus different request or authority fails `409`.
7. Duplicate tenant slug fails with a distinct sanitized resource-conflict code, including under concurrent application creates.
8. In-memory, unavailable and SQLAlchemy adapters implement one creator protocol.
9. The response is `201`, typed in OpenAPI, and includes exact `Location` and `ETag` headers without implying approval.
10. Cross-tenant, campaign-scoped grant, wrong purpose/resource/action, absent/revoked-effective grant and adapter-scope-leak cases fail closed before persistence.
11. Unit, API, BOLA, idempotency, atomicity, PostgreSQL/RLS, lint, format, mypy, coverage, program, eval, safety and secret gates pass.
12. Production remains `BLOCKED`; no outbox consumer may perform an external political effect.
