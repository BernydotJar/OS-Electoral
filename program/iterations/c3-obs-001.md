## Objective

Add an operational control plane for the API and internal worker, plus a native PostgreSQL backup/restore verification path, without creating an environment, deploying, contacting citizens, profiling voters, publishing, spending, changing grants or invoking an external campaign provider.

## Implemented boundary

- sanitized JSON logs with an allowlist of operational fields;
- validated correlation IDs and W3C trace-context continuation with a new server span;
- low-cardinality Prometheus metrics for API requests, readiness and database pool state;
- bearer-protected metrics outside development/test, with fail-closed shared-environment configuration;
- atomic Prometheus textfiles for the internal outbox worker and recovery verifier;
- alert rules for readiness, HTTP errors, latency, dead letters, stale backups and stale restore verification;
- native PostgreSQL custom-format backup verification limited to an isolated `*_test` source;
- target database names restricted to a distinct `*_restore_test` database;
- immutable PostgreSQL client-image pinning and client/server major-version equality;
- archive catalog validation, isolated restore, Alembic revision comparison and all-public-table row-count comparison;
- SHA-256 manifest, owner-only backup/manifest permissions and unconditional restore-database cleanup;
- a dedicated PostgreSQL 18 CI job that migrates, seeds, backs up, restores, compares, cleans up and retains evidence;
- an operations runbook that keeps managed backups, OTLP export, dashboards, routing, RPO/RTO and staging failure exercises explicitly pending.

## Safety properties

1. Metrics never use tenant, campaign, principal, citizen or voter identifiers as labels.
2. Logs do not include authorization headers, tokens, request bodies, arbitrary URLs or exception messages.
3. The metrics scrape token is operational-only and creates no product authority.
4. The recovery verifier rejects a production-like source name and cannot target the source database.
5. No external delivery transport was added to the worker.
6. The repository job is test evidence; it is not a production backup or a staging disaster-recovery exercise.
7. Production remains `BLOCKED` until managed-environment, security, privacy, recovery and human gates pass.
